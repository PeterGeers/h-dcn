# SAM Circular Dependency Fix - Bugfix Design

## Overview

The `sam deploy --stack-name h-dcn` deployment fails because `CreateOrderFunction` has an environment variable (`MOLLIE_WEBHOOK_URL`) that uses `!Sub "https://${MyApi}.execute-api..."`, creating a CloudFormation reference from the Lambda function back to the API Gateway resource. Since the API stage already depends on all Lambda functions through their API event integrations, SAM's auto-generated `*PermissionStage` resources form a circular dependency chain that prevents deployment.

The fix removes the `${MyApi}` reference from the environment variable and instead constructs the webhook URL at runtime using known, stable values (region, stack name / API ID via SSM or hardcoded domain), or uses `!Sub` with only parameters that don't create circular references.

## Glossary

- **Bug_Condition (C)**: The condition that triggers the bug — a CloudFormation `!Sub` expression in a Lambda Environment block that references the `MyApi` resource, creating a dependency cycle with PermissionStage resources
- **Property (P)**: The desired behavior — deployment succeeds and the webhook URL is still valid and reachable
- **Preservation**: Existing Mollie webhook routing, API Gateway permissions, and payment flow must remain unchanged
- **MyApi**: The `AWS::Serverless::Api` resource in `backend/template.yaml` that defines the REST API
- **PermissionStage**: Auto-generated `AWS::Lambda::Permission` resources SAM creates to allow API Gateway to invoke Lambda functions for the deployed stage
- **CreateOrderFunction**: Lambda function at `backend/handler/create_order/app.py` that creates orders and initiates Mollie payments
- **MollieWebhookFunction**: Lambda function at `backend/handler/mollie_webhook/app.py` that receives payment status callbacks from Mollie at `/mollie-webhook`

## Bug Details

### Bug Condition

The bug manifests when the SAM template contains a `!Sub` expression in a Lambda function's `Environment.Variables` that references the `MyApi` resource. This creates a bidirectional dependency: the Lambda function depends on `MyApi` (via the environment variable), and the API stage depends on the Lambda function (via the API event integration). SAM's auto-generated PermissionStage resources amplify this into a full circular dependency chain involving all functions attached to the API.

**Formal Specification:**

```
FUNCTION isBugCondition(template)
  INPUT: template of type SAMTemplate
  OUTPUT: boolean

  FOR EACH function IN template.Resources WHERE function.Type == "AWS::Serverless::Function"
    FOR EACH envVar IN function.Properties.Environment.Variables
      IF envVar.value CONTAINS !Sub OR !Ref referencing "MyApi"
         AND function.Properties.Events CONTAINS event WHERE event.Type == "Api"
                AND event.Properties.RestApiId == !Ref MyApi
      THEN RETURN true
    END FOR
  END FOR
  RETURN false
END FUNCTION
```

### Examples

- **Current (fails)**: `MOLLIE_WEBHOOK_URL: !Sub "https://${MyApi}.execute-api.${AWS::Region}.amazonaws.com/${Environment}/mollie-webhook"` — creates circular dependency, deployment fails
- **Fixed (succeeds)**: `MOLLIE_WEBHOOK_URL: !Sub "https://${MyApi.RootResourceId}..."` — still fails (same reference)
- **Fixed (succeeds)**: Construct URL without referencing `MyApi` at all — e.g. use a hardcoded domain, an SSM parameter, or build the URL at runtime in Python — deployment succeeds
- **Edge case**: Other functions using `RestApiId: !Ref MyApi` in their Events section (not Environment) — this is fine and does NOT create a cycle

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**

- Mollie webhook callbacks to `https://<api-id>.execute-api.eu-west-1.amazonaws.com/<stage>/mollie-webhook` must continue to reach the `MollieWebhookFunction`
- All Lambda functions with API events must continue to have correct invoke permissions (PermissionStage resources auto-generated by SAM)
- The `CreateOrderFunction` must continue to pass a valid, reachable webhook URL to the Mollie API when creating payments
- Other Lambda functions that reference `!Ref MyApi` only in their `Events.*.Properties.RestApiId` must continue to work (these do not cause circular dependencies)

**Scope:**
All inputs that do NOT involve the `CreateOrderFunction`'s `MOLLIE_WEBHOOK_URL` environment variable referencing `MyApi` should be completely unaffected by this fix. This includes:

- All other Lambda function definitions and their environment variables
- API Gateway route definitions and method configurations
- IAM roles and policies
- DynamoDB table references
- All other `!Sub` expressions that don't reference `MyApi` from within a Lambda Environment block

## Hypothesized Root Cause

Based on the bug description, the root cause is clear and singular:

1. **Direct `!Sub` reference to `MyApi` in Lambda Environment block**: The line `MOLLIE_WEBHOOK_URL: !Sub "https://${MyApi}.execute-api.${AWS::Region}.amazonaws.com/${Environment}/mollie-webhook"` in `CreateOrderFunction`'s environment creates a CloudFormation dependency from the Lambda function resource to the `MyApi` resource.

2. **Bidirectional dependency via SAM transform**: The same `CreateOrderFunction` has an API event (`RestApiId: !Ref MyApi`), which makes the API stage depend on the function. SAM generates `*PermissionStage` resources for ALL functions attached to the API, and the stage deployment resource depends on all of them. This creates: `MyApi Stage → PermissionStage(all functions) → Lambda Functions → MyApi` — a cycle.

3. **Amplification effect**: Even though only `CreateOrderFunction` has the problematic environment variable, the circular dependency error lists PermissionStage resources for many other functions (e.g., `GetMembersFilteredFunctionGetMembersFilteredPermissionStage`) because the SAM-generated stage deployment depends on ALL permission resources for all functions in the API.

## Correctness Properties

Property 1: Bug Condition - Deployment Succeeds Without Circular Dependency

_For any_ SAM template where `CreateOrderFunction`'s `MOLLIE_WEBHOOK_URL` environment variable does NOT contain a `!Sub` or `!Ref` referencing the `MyApi` resource, the `sam deploy` command SHALL complete successfully without CloudFormation circular dependency errors, and all Lambda functions and API Gateway routes SHALL be deployed correctly.

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - Webhook URL Remains Valid and Reachable

_For any_ Mollie payment created by `CreateOrderFunction`, the webhook URL passed to the Mollie API SHALL resolve to a valid API Gateway endpoint that routes to the `MollieWebhookFunction`, preserving the existing payment callback flow and ensuring payment status updates are received.

**Validates: Requirements 3.1, 3.3**

Property 3: Preservation - SAM Auto-Generated Permissions Unchanged

_For any_ Lambda function with an API event attached to `MyApi`, the SAM transform SHALL continue to generate correct `*PermissionStage` resources granting API Gateway invoke permissions, preserving all existing API Gateway integrations.

**Validates: Requirements 3.2, 3.4**

## Fix Implementation

### Changes Required

The fix removes the `${MyApi}` reference from the `MOLLIE_WEBHOOK_URL` environment variable. The recommended approach is to construct the URL using only parameters and pseudo-parameters that don't reference the API resource itself.

**File**: `backend/template.yaml`

**Function**: `CreateOrderFunction` Environment block

**Specific Changes**:

1. **Replace `${MyApi}` with a constructed domain**: Instead of referencing the API resource, use `!Sub` with `${AWS::StackName}` and known conventions, or better yet, use a hardcoded/parameterized base URL since the API domain is predictable after first deployment.

   **Option A — Use SSM parameter (most robust, requires one-time manual setup):**

   ```yaml
   MOLLIE_WEBHOOK_URL: !Sub "{{resolve:ssm:/h-dcn/api-base-url}}/mollie-webhook"
   ```

   Store the API base URL in SSM after first deployment.

   **Option B — Construct at runtime in Python (no template changes needed beyond removing the reference):**

   ```yaml
   # Remove MOLLIE_WEBHOOK_URL from template entirely, or set to empty
   # Let the Python handler construct it at runtime
   API_GATEWAY_ID: "" # Not needed if using approach below
   ```

   Then in `app.py`, construct the URL from the API Gateway context available in the Lambda event.

   **Option C — Hardcode the known API URL (simplest, works for single-environment):**

   ```yaml
   MOLLIE_WEBHOOK_URL: !Sub "https://${AWS::StackName}.execute-api.${AWS::Region}.amazonaws.com/${Environment}/mollie-webhook"
   ```

   Note: This won't work because `${AWS::StackName}` is the stack name, not the API ID.

   **Option D (Recommended) — Construct URL at runtime from the event's API Gateway context:**
   Remove `MOLLIE_WEBHOOK_URL` from the SAM template environment variables and have the Python handler build it from `event['requestContext']` which contains the API ID and stage:

   ```python
   api_id = event['requestContext']['apiId']
   stage = event['requestContext']['stage']
   region = os.environ.get('AWS_REGION', 'eu-west-1')
   webhook_url = f"https://{api_id}.execute-api.{region}.amazonaws.com/{stage}/mollie-webhook"
   ```

2. **Update `backend/handler/create_order/app.py`**: Modify the handler to construct the webhook URL from the request context instead of relying on the environment variable.

3. **Keep fallback**: Retain `os.environ.get('MOLLIE_WEBHOOK_URL', '')` as a fallback/override mechanism for local testing, but the primary URL construction should come from the event context.

4. **Remove or comment the problematic line in template.yaml**: The line `MOLLIE_WEBHOOK_URL: !Sub "https://${MyApi}.execute-api.${AWS::Region}.amazonaws.com/${Environment}/mollie-webhook"` must be removed or replaced with a value that does not reference `MyApi`.

5. **No changes to MollieWebhookFunction**: The webhook handler's route definition (`Path: /mollie-webhook`) remains unchanged — only the URL construction in `CreateOrderFunction` changes.

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, confirm the circular dependency exists in the current template, then verify the fix resolves it while preserving webhook functionality.

### Exploratory Bug Condition Checking

**Goal**: Confirm that the current template has the circular dependency and understand the exact error. This validates our root cause analysis.

**Test Plan**: Attempt `sam build` and `sam deploy --no-execute-changeset` on the unfixed template to observe the circular dependency error. Alternatively, use `cfn-lint` or `sam validate` to detect the issue statically.

**Test Cases**:

1. **Deploy test (unfixed)**: Run `sam deploy` with current template — should fail with circular dependency error
2. **Dependency analysis**: Trace the reference chain: `CreateOrderFunction → MyApi` (via `!Sub`) and `MyApi Stage → CreateOrderFunction` (via Events) to confirm the cycle
3. **Scope confirmation**: Verify that other functions (e.g., `GetMembersFilteredFunction`) appear in the error even though they don't have the problematic env var — confirms the amplification effect

**Expected Counterexamples**:

- CloudFormation error: "Circular dependency between resources: [GetMembersFilteredFunctionGetMembersFilteredPermissionStage, ...]"
- The cycle is caused solely by the `MOLLIE_WEBHOOK_URL` reference to `${MyApi}`

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds (template with `MyApi` reference removed), the deployment succeeds.

**Pseudocode:**

```
FOR ALL template WHERE NOT containsMyApiRefInEnvVar(template) DO
  result := samDeploy(template)
  ASSERT result.status == "SUCCESS"
  ASSERT result.circularDependencyError == false
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold (non-deployment behavior), the fixed code produces the same results.

**Pseudocode:**

```
FOR ALL request WHERE request.type == "POST /orders" DO
  webhookUrl_fixed := constructWebhookUrl(request.event)
  ASSERT webhookUrl_fixed == "https://<api-id>.execute-api.eu-west-1.amazonaws.com/<stage>/mollie-webhook"
  ASSERT molliePayment.webhookUrl == webhookUrl_fixed
  ASSERT webhookUrl_fixed routes to MollieWebhookFunction
END FOR
```

**Testing Approach**: Unit testing is most appropriate here because:

- The fix is localized to URL construction logic
- We can mock the Lambda event context to verify URL construction
- Integration testing via actual deployment confirms the circular dependency is resolved
- Property-based testing can generate various event contexts to verify URL construction is always correct

**Test Plan**: Write unit tests that verify the webhook URL is correctly constructed from the event's request context, then deploy to confirm the circular dependency is resolved.

**Test Cases**:

1. **Webhook URL construction**: Verify that given a Lambda event with `requestContext.apiId` and `requestContext.stage`, the correct webhook URL is produced
2. **Mollie payment creation**: Verify the webhook URL passed to Mollie matches the expected format
3. **Fallback behavior**: Verify that if `MOLLIE_WEBHOOK_URL` env var is set (e.g., in tests), it takes precedence over runtime construction
4. **MollieWebhookFunction routing**: Verify the `/mollie-webhook` route still maps to the correct handler after deployment

### Unit Tests

- Test webhook URL construction from various `event['requestContext']` values
- Test fallback to environment variable when `requestContext` is unavailable
- Test that Mollie payment creation uses the correctly constructed URL
- Test edge cases: missing `apiId`, missing `stage`, different regions

### Property-Based Tests

- Generate random API IDs and stage names, verify constructed URL always matches the pattern `https://{apiId}.execute-api.{region}.amazonaws.com/{stage}/mollie-webhook`
- Generate various Lambda event structures, verify URL construction is robust and never produces invalid URLs
- Verify that for any event context, the constructed URL always ends with `/mollie-webhook`

### Integration Tests

- Deploy the fixed template with `sam deploy` and verify no circular dependency error
- After deployment, create a test order and verify the webhook URL in the Mollie payment request is correct
- Trigger a test webhook callback to verify `MollieWebhookFunction` still receives it correctly
- Verify all other API routes continue to function after deployment
