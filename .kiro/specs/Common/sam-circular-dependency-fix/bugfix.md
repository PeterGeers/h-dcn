# Bugfix Requirements Document

## Introduction

The SAM backend deployment (`sam deploy --stack-name h-dcn`) fails with a CloudFormation circular dependency error. The cycle involves dozens of `*PermissionStage` resources — these are auto-generated by SAM to grant API Gateway permission to invoke Lambda functions for the deployed stage.

The root cause is that the `CreateOrderFunction` Lambda function has an environment variable (`MOLLIE_WEBHOOK_URL`) that references `MyApi` via `!Sub "https://${MyApi}.execute-api..."`. This creates a dependency from the Lambda function back to the API resource, while the API stage already depends on the Lambda function through the API event integration. SAM's auto-generated permission resources create a cycle because the stage deployment cannot resolve its dependency order.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN `CreateOrderFunction` environment variable `MOLLIE_WEBHOOK_URL` uses `!Sub "https://${MyApi}.execute-api.${AWS::Region}.amazonaws.com/${Environment}/mollie-webhook"` THEN CloudFormation detects a circular dependency between PermissionStage resources and the deployment fails

1.2 WHEN `sam deploy` is executed THEN the stack update/create is rejected with error "Circular dependency between resources: [GetMembersFilteredFunctionGetMembersFilteredPermissionStage, ...]" and no resources are deployed

### Expected Behavior (Correct)

2.1 WHEN `CreateOrderFunction` needs a webhook URL for Mollie THEN the system SHALL construct the URL without creating a direct CloudFormation reference from the Lambda function to the API Gateway resource, breaking the circular dependency

2.2 WHEN `sam deploy` is executed THEN the stack update/create SHALL succeed without circular dependency errors and all Lambda functions and API Gateway routes SHALL be deployed correctly

### Unchanged Behavior (Regression Prevention)

3.1 WHEN Mollie sends a webhook callback to the configured URL THEN the system SHALL CONTINUE TO route the request to the `MollieWebhookFunction` Lambda handler at the `/mollie-webhook` path

3.2 WHEN any Lambda function with an API event is deployed THEN the system SHALL CONTINUE TO have correct API Gateway invoke permissions (PermissionStage resources) auto-generated by SAM

3.3 WHEN the `CreateOrderFunction` creates a Mollie payment THEN the system SHALL CONTINUE TO pass a valid, reachable webhook URL to the Mollie API so payment status callbacks are received

3.4 WHEN other Lambda functions reference `!Ref MyApi` in non-Environment contexts (such as the `RestApiId` property in Events) THEN the system SHALL CONTINUE TO work correctly as these do not create circular dependencies
